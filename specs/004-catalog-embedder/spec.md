# Feature Specification: Catalog Auto-Embedding Service

**Feature Branch**: `feat/recommend-catalog-embedder`
**Created**: 2026-05-09
**Status**: Draft
**Input**: User description: "Catalog feature for auto-embedding raw client content via separate cmd/embedder binary — remove the requirement that clients (e.g. social-network backends) precompute their own dense embeddings. Approach A from prior discussion: a new dedicated binary, decoupled from the data plane, that consumes raw items and produces dense vectors that flow into the existing dense Qdrant collections."

## Clarifications

### Session 2026-05-09

- Q: Embedding algorithm class for V1 → A: Deterministic, training-free in-process Go algorithm (hashing-trick / TF-IDF + dimension reduction). V1 ships at least one such built-in strategy and depends on no external embedding model service. V1 architecture MUST additionally preserve a clean plug-in seam so that external LLM-based embedding strategies (OpenAI-, Cohere-, self-hosted-LLM-style) can be added in a follow-up phase without changing the catalog ingest contract, the catalog data model, or operator-visible surfaces.
- Q: Strategy-version transition semantics — what happens to new catalog ingests while a namespace's strategy version is bumping or being re-embedded? → A: New ingests are embedded under the new active strategy version *immediately* on bump; existing items keep their old-version tag and old vector until the operator-triggered re-embed reaches them. The dense collection is therefore explicitly mixed-version during the transition window, and the recommend service is expected to tolerate this since dense scores remain numerically valid for the deterministic V1 strategy. The "active strategy version" of a namespace is whatever new ingests are being written under at any moment.
- Q: Text tokenization scope for V1's built-in deterministic strategy → A: Whitespace tokenization + character n-grams (n in 3..5), language-agnostic. V1 ships no per-language segmenter and no shipped dictionary; Vietnamese is supported via the n-gram fallback rather than a Vietnamese-aware word segmenter. Per-language tokenizers (including a dictionary-based Vietnamese segmenter) are explicitly out of scope for V1 but may be added later as a new strategy version (or, once V2 ships, as a separate strategy entirely) — both pathways trigger the standard namespace-wide re-embed flow.
- Q: Catalog ingest payload shape — what does the client send and what gets embedded? → A: The client sends a single dedicated `content` text field that is the only input fed to the embedder. Optional structured metadata may be attached per item, but is stored alongside as opaque associated data and does not contribute to either the embedding or the per-item content hash. The content hash is therefore a pure function of the `content` field, making idempotency (FR-002, FR-011) unambiguous. Future external-LLM strategies receive the same single `content` field with no per-strategy reshaping.
- Q: V1 operator workflow to enable catalog auto-embedding for a namespace → A: The operator enables the feature AND explicitly selects a strategy identifier+version (in V1 there is exactly one valid value, the built-in deterministic strategy; explicit selection is still required so the audit trail is unambiguous and so the UX matches the V2 multi-strategy world). The produced vector dimension is derived from the chosen strategy version and MUST match the namespace's existing `embedding_dim` field; if they do not match, enabling fails with a clear, operator-facing error. There is no separate per-namespace dimension knob for catalog — `embedding_dim` remains the single source of truth for vector dimension across both BYOE and catalog paths.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Social-network client ingests raw posts and immediately benefits from hybrid recommendations (Priority: P1)

A backend engineer at a social-network product wants Codohue to recommend posts to users. Today they would have to stand up their own embedding pipeline (host a model or call a third-party API, manage retries, batching, cost, model versioning) and only then push dense vectors into Codohue via the BYOE endpoints. They want to instead push *the raw post content* — text plus a few metadata fields — and let Codohue produce the embedding so that the post becomes discoverable through hybrid (sparse + dense) recommendations without further work on their side.

**Why this priority**: This is the core value of the feature. Every new integrator that does not already operate an embedding stack pays a high adoption tax under BYOE. Lowering that bar is the entire reason for the feature; without this story none of the others matter.

**Independent Test**: Can be fully tested by ingesting a stream of representative posts through the catalog endpoint for a single namespace, waiting for the asynchronous embedding work to drain, and confirming that (a) the namespace's dense object index contains a vector for each ingested object, and (b) recommendations served for a user who has interacted with one of those posts include other catalog-only items via the dense leg of the hybrid blend.

**Acceptance Scenarios**:

1. **Given** a namespace configured to use the catalog auto-embedding feature with the V1 built-in embedding strategy, **When** a client posts a new catalog item containing a `namespace`-scoped object id and raw content, **Then** the call returns an accepted response without blocking on embedding work.
2. **Given** a catalog item has been accepted, **When** the embedding work has drained (within the namespace's stated freshness target), **Then** querying that object via the recommendation flow surfaces it for an appropriate subject and the namespace's dense object index contains a non-zero vector for the object.
3. **Given** a catalog item already exists for a namespace and object id, **When** the client re-ingests the same content (same content hash) under the same embedding strategy version, **Then** the system treats the call as a no-op and does not redo embedding work.
4. **Given** a catalog item already exists, **When** the client re-ingests with substantively different content (different content hash), **Then** the system re-embeds the item and the namespace's dense object index is updated to reflect the new vector.
5. **Given** a client is using the existing BYOE write path for the same namespace, **When** they later switch that namespace to catalog auto-embedding, **Then** previously stored BYOE vectors continue to serve recommendations until they are superseded by catalog-driven embeddings, with no recommendation-side outage.

### User Story 2 - Operator enables and configures the embedding strategy per namespace (Priority: P2)

An operator running Codohue needs, per namespace, to (a) enable catalog auto-embedding and (b) explicitly select an embedding strategy identifier and version. The vector dimension is NOT a separate knob: it is derived from the chosen strategy version and must match the namespace's existing `embedding_dim` field, otherwise enabling fails. In V1 there is exactly one valid strategy (the built-in deterministic one), but explicit selection is still required so the audit trail is unambiguous and so that adding a future external-LLM strategy (e.g. an OpenAI-style provider with credentials and a model identifier) does not require redesigning the operator UX, the persisted configuration shape, or the catalog data model.

**Why this priority**: Operationally critical so that namespaces have explicit, observable strategy state and so that the V2 plug-in path is a value-only addition rather than a redesign. Not required for the very first end-to-end demo, since there is only one strategy in V1.

**Independent Test**: Can be fully tested by enabling catalog auto-embedding on a namespace through the admin surface, ingesting catalog items, and confirming that (a) the namespace records its strategy identifier and version with each embedded item, (b) the produced vector dimension matches the namespace's configured dense embedding dimension, and (c) ingesting into a namespace that has not been enabled is rejected with a clear error.

**Acceptance Scenarios**:

1. **Given** an operator is logged into the admin surface and a namespace has its `embedding_dim` already set, **When** they enable catalog auto-embedding for that namespace and explicitly select a strategy identifier+version whose natural output dimension equals the namespace's `embedding_dim`, **Then** the configuration is accepted and subsequent catalog items for that namespace are embedded by the selected strategy, with the namespace recording the active strategy identifier and version.
2. **Given** an operator attempts to enable catalog auto-embedding for a namespace, **When** the explicitly selected strategy version's natural output dimension does not match the namespace's existing `embedding_dim`, **Then** the enabling call is rejected with a clear, operator-facing error that names both dimensions; the namespace remains in its pre-call state and no catalog records are created.
3. **Given** an operator has not enabled catalog auto-embedding for a namespace, **When** a client posts a catalog item to that namespace, **Then** the catalog ingest call is rejected with a clear error indicating the namespace is not enabled for auto-embedding, and BYOE clients are unaffected.
4. **Given** the V1 operator UX and persisted configuration shape, **When** a future release adds an external-LLM strategy that introduces credentials and provider identifiers, **Then** the new strategy slots into the same UX/config/data-model surfaces with additive fields only; no migration of existing namespace records or catalog records is required for namespaces that stay on the built-in strategy.

### User Story 3 - Operator triggers a namespace-wide re-embedding when the embedding strategy or its version changes (Priority: P3)

An operator changes the active embedding strategy or its version for a namespace (e.g. upgrading the built-in strategy from `v1` to `v2`, or — once V2 ships — switching from the built-in strategy to an external LLM strategy). Existing dense vectors in that namespace were produced under the previous strategy/version and are not directly comparable to vectors that would be produced going forward. The operator needs a way to refresh every catalog item's embedding under the new strategy/version without taking the recommendation surface offline and without requiring clients to re-send raw content.

**Why this priority**: Real but episodic — needed whenever a namespace's strategy or version changes, but not on the critical path for first integration. Without it, strategy/version changes would force clients to redo ingestion, which contradicts the purpose of the feature. Also serves as the operational seam through which V2 LLM upgrades will roll out.

**Independent Test**: Can be fully tested by populating a namespace with N catalog items, bumping its embedding strategy version (or — when V2 ships — switching to a different strategy), triggering the namespace-level re-embed action, and confirming that (a) every item is re-embedded under the new strategy/version, (b) recommendations continue to be served throughout the operation, and (c) the operator can observe progress (items remaining, items completed, items failed).

**Acceptance Scenarios**:

1. **Given** a namespace has catalog items embedded under strategy version `v1`, **When** the operator bumps the namespace's active strategy version to `v2` and triggers a namespace-wide re-embed, **Then** every catalog item is enqueued for re-embedding under `v2` and items are marked with the new strategy/version on completion.
2. **Given** a namespace's active strategy version has just been bumped from `v1` to `v2`, **When** a new catalog item is ingested before the re-embed for that namespace has completed, **Then** the new item is embedded under `v2` immediately (not held back, not embedded under `v1`), and during this transition window the namespace's dense object index legitimately contains a mix of `v1`-tagged and `v2`-tagged points.
3. **Given** a namespace-wide re-embed is in progress, **When** a client requests recommendations for a subject in that namespace, **Then** recommendations continue to be served (using whichever vectors are currently present) without error.
4. **Given** a namespace-wide re-embed has been triggered, **When** the operator inspects progress through the admin surface, **Then** they see backlog size, in-flight count, completed count under the new strategy/version, and any items currently in the failed/dead-letter state.
5. **Given** a re-embed is partially complete, **When** the operator pauses or cancels the operation, **Then** items already re-embedded retain their new-version vectors and items not yet processed remain on the previous strategy/version until the next trigger; new ingest during this paused state continues under the active (post-bump) version.

### Edge Cases

- **Embedder offline / backlog growth**: The catalog ingest path must continue to accept new items even when no embedder instances are running. The backlog must be observable so operators can react before storage pressure becomes an outage.
- **Transient embedding failure**: Embedding attempts that fail (e.g. malformed content, internal panic, future external-strategy network/transient errors) MUST retry with backoff up to a configurable maximum before being moved to a dead-letter state that operators can inspect and re-drive. The retry/dead-letter contract MUST be uniform across the V1 built-in strategy and any future externally-backed strategy.
- **Oversized content**: Content that exceeds the strategy's input-length limit must be either truncated by an explicit policy or rejected at ingest time — never silently discarded after acceptance.
- **Dimension mismatch**: If the embedding strategy produces a vector whose dimension differs from the namespace's configured dense dimension, the system must refuse to write it rather than corrupt the dense collection.
- **Object simultaneously written via BYOE and catalog**: The system must define a single source of truth per object so that one path does not silently overwrite the other on every batch cycle.
- **Object deletion**: Deleting an object must remove both its catalog record and its dense vector, consistent with the existing object-deletion guarantees on the client API.
- **Namespace not enabled for auto-embedding**: Catalog ingest calls for such a namespace must fail closed with a clear message so that misconfigured clients cannot silently fill storage with un-embeddable content.
- **Re-embed during scheduled batch recompute**: A namespace-wide re-embed running concurrently with the existing scheduled batch recompute must not produce inconsistent dense state.
- **Mixed-version dense collection during transition**: When a namespace's active strategy version is bumped, the dense object index is expected to legitimately contain a mix of points tagged with the old and new versions until the operator-triggered re-embed completes. The system MUST record the strategy identifier and version associated with each catalog item so this mixed state is observable and re-embed completion is detectable, and MUST NOT consider the mixed state an error in itself.
- **Forward-compat regression**: A V1 namespace using the built-in strategy MUST continue to function unchanged when a follow-up release adds new external-LLM strategies — no automatic migration, no implicit re-embed, no operator-visible surface change for that namespace.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST expose a catalog ingest capability that accepts, per namespace, a raw catalog item identified by a stable object id and carrying exactly one mandatory text field (`content`) that is the embedding input, plus optional structured metadata that is stored alongside as opaque associated data and does NOT feed the embedder. The call is authenticated by the namespace's existing API key.
- **FR-002**: System MUST persist each accepted catalog item with a content hash computed exclusively over the `content` field (metadata is excluded), so that resubmission of identical `content` for the same object id (under the same active strategy version) is idempotent and does not redo embedding work, regardless of whether the metadata changed.
- **FR-003**: System MUST accept catalog ingest calls without blocking on embedding work — the call returns immediately and the embedding is produced asynchronously.
- **FR-004**: System MUST process catalog items in a separate, independently deployable component so that the data plane (event ingest, recommendations) is not impacted by embedding latency, throughput, or failures.
- **FR-005**: System MUST upsert produced dense vectors into the namespace's existing dense object collection so that they participate in the existing hybrid recommendation flow with no recommend-side change required.
- **FR-006**: System MUST support deployments that disable the catalog feature entirely — namespaces using only BYOE MUST continue to function with no degradation when no embedder component is running.
- **FR-007**: System MUST allow operators to enable catalog auto-embedding per namespace through the admin surface. Enabling MUST require an explicit strategy identifier+version selection (in V1 the only valid value is the built-in deterministic strategy), and the namespace MUST record the active strategy identifier and version. The persisted strategy profile MUST carry an extension slot for strategy-specific parameters (e.g. future credentials references) so that introducing additional strategies in a follow-up phase is purely additive. The vector dimension is NOT a separate knob: it is derived from the selected strategy version and MUST match the namespace's existing `embedding_dim` field.
- **FR-008**: System MUST reject catalog ingest calls for namespaces that have not been enabled for auto-embedding, with a clear error distinguishing this from "object not found" or generic auth failures.
- **FR-009**: System MUST validate that any vector produced by the active embedding strategy matches the namespace's existing `embedding_dim` and refuse to write mismatched vectors. Dimension validation MUST occur both at enable time (rejecting the operator's enable call) and at runtime (rejecting any individual vector that fails to match), so that a strategy version misregistered as producing dim X but actually producing dim Y cannot corrupt the dense object index.
- **FR-010**: System MUST retry failed embedding attempts with backoff up to a configurable maximum, after which the item moves to a dead-letter state visible to operators. The retry / dead-letter contract MUST be defined uniformly across embedding strategies so that future external-LLM strategies inherit the same operator semantics.
- **FR-011**: System MUST detect and skip re-embedding when an item's content hash and the namespace's active strategy identifier+version are both unchanged from the last successful embedding.
- **FR-012**: System MUST re-embed an item when its content changes (new content hash) and update the dense vector accordingly.
- **FR-013**: System MUST allow operators to trigger a namespace-wide re-embed when the strategy identifier or strategy version changes, without taking recommendations offline during the operation.
- **FR-014**: System MUST surface, per namespace, the catalog backlog size, in-flight count, success count, and dead-letter count to operators (admin surface and operational metrics).
- **FR-015**: System MUST surface, per namespace, operational embedding throughput and resource indicators (items embedded per period, p95 embed latency per item, and a strategy-specific work-volume indicator — e.g. tokens processed for token-based strategies) sufficient for operators to detect runaway clients and capacity-plan the embedder. The set of indicators MUST be defined uniformly so future external-LLM strategies can additionally report cost / billed-units through the same surface without redesign.
- **FR-016**: System MUST allow operators to inspect and re-drive items in the dead-letter state.
- **FR-017**: System MUST remove the catalog record and the corresponding dense vector when the existing object-deletion path is invoked for that object.
- **FR-018**: System MUST define a single source of truth per `(namespace, object id)` so that catalog-driven and BYOE-driven dense writes do not silently overwrite each other on every cycle. The chosen policy MUST be documented and observable per namespace.
- **FR-019**: System MUST produce structured operational logs and metrics for embedding runs, consistent with the existing batch-run history surfaces, so the admin dashboard can render embedder activity using the same operator mental model.
- **FR-020**: System MUST handle oversized content by an explicit, configurable policy (truncate or reject at ingest) — never silently discard after acceptance.
- **FR-021**: System MUST encapsulate the embedding algorithm behind a stable internal seam (a strategy abstraction) so that adding a future external-LLM strategy is a purely additive change. Specifically, V2 strategies MUST be addable without (a) altering the catalog ingest API contract, (b) altering the catalog item or strategy-profile data model in non-additive ways, (c) altering operator-visible admin endpoints in non-additive ways, or (d) requiring re-embedding of namespaces that remain on a previously-active strategy.

### Key Entities

- **Catalog Item**: The raw, unembedded record for an object in a namespace. Holds the `content` text field (the embedder's only input), optional structured metadata stored alongside but not embedded, content hash computed over `content` only, the strategy identifier and version under which it was last embedded, the timestamp it was last embedded, and its current lifecycle state (pending, in-flight, embedded, failed, dead-letter). Uniquely identified by `(namespace, object id)`.
- **Embedding Strategy Profile**: Per-namespace configuration that selects which embedding strategy is active, with which strategy version, and at which expected output dimension. In V1 the only available strategy is the built-in deterministic strategy. The profile carries an extension slot for strategy-specific parameters (e.g. future credentials references, future remote model identifiers, future per-strategy length limits) so that introducing external-LLM strategies in a follow-up phase is a purely additive change. Mutating the strategy identifier or version is the trigger for namespace-wide re-embed.
- **Embedding Job**: A unit of pending embedding work for a single catalog item, tracking attempt count, last error, and the timestamp it was claimed by an embedder worker. Used for retry, backoff, and dead-letter accounting. Uniform across strategies.
- **Embedding Run Record**: A summary record analogous to the existing `batch_run_logs` surface, capturing each embedder activity period (window covered, items embedded, items failed, strategy identifier, strategy version, log lines) so that operators can inspect activity and debug regressions through the same admin views they already use for cron runs.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A new client team integrating a social-network-style product can move from zero to having catalog items appearing in recommendations in under one engineering day, without standing up any embedding infrastructure of their own and without depending on any external embedding service in V1.
- **SC-002**: For a namespace operating within the embedder's nominal throughput envelope, 95% of accepted catalog items become discoverable through dense recommendations within 60 seconds of acceptance.
- **SC-003**: 99% of accepted catalog items reach an embedded state without operator intervention; the remaining 1% are visible in the dead-letter state with clear failure context.
- **SC-004**: Catalog ingest sustains its target throughput without degrading recommendation tail latency by more than 5% at p99 under steady-state load.
- **SC-005**: Disabling the catalog feature (no embedder instances running, no namespace enabled for auto-embedding) MUST have zero impact on existing BYOE clients — measured by an unchanged recommendation success rate and unchanged tail latency for those namespaces.
- **SC-006**: A namespace-wide re-embed of 10,000 items completes without serving any recommendation error attributable to the re-embed, and operator-visible progress reaches 100% within an operator-observable window.
- **SC-007**: Per-namespace embedding throughput / resource indicators are visible to operators within 5 minutes of accrual, sufficient to detect a runaway client (sustained ingest rate consuming a disproportionate share of embedder capacity) before recommendation tail latency or backlog drift triggers user-visible degradation.
- **SC-008**: Operators can re-drive every item in the dead-letter state from the admin surface in a single action, and successful re-drives are reflected in the namespace's dense object index without manual database intervention.
- **SC-009**: A future external-LLM strategy can be added to the codebase as a purely additive change — no catalog-item, strategy-profile, ingest-API, or operator-endpoint contract is broken; namespaces that remain on the V1 built-in strategy require neither configuration migration nor re-embedding.

## Assumptions

- **Content modality**: V1 supports textual content only. Image, audio, and other multimodal embeddings are explicitly out of scope and will be addressed in a follow-up feature if needed.
- **Language scope**: V1 declares language-agnostic support: any text content can be ingested and embedded by the built-in strategy via whitespace tokenization plus character n-grams (n in 3..5). V1 makes no per-language quality guarantee and ships no per-language segmenter or dictionary. The primary target use case (Vietnamese social-network posts) is supported via the n-gram fallback rather than a Vietnamese-aware word segmenter; clients requiring higher per-language quality MUST wait for a follow-up strategy version that introduces language-specific preprocessing.
- **Embedding strategy hosting**: V1 ships exactly one built-in, in-process, deterministic embedding strategy implemented in Go (hashing-trick / TF-IDF + dimension reduction class). V1 does not call any external embedding model service. External LLM-based embedding strategies (OpenAI-, Cohere-, self-hosted-LLM-style) are explicitly out of scope for V1 *as implementations* but are explicitly *in scope as a forward-compatibility constraint* — the strategy abstraction, configuration shape, data model, and admin surfaces MUST be designed so that adding such strategies later is a purely additive change.
- **Subject (user) embeddings**: Subject-side dense vectors continue to be derived from object-side vectors by the existing cron mean-pooling step. The catalog auto-embedding feature only produces *object-side* dense vectors.
- **Per-namespace opt-in**: Catalog auto-embedding is opt-in per namespace. A namespace that is not explicitly enabled continues to behave exactly as today (BYOE or sparse-only).
- **Source-of-truth precedence**: When catalog auto-embedding is enabled for a namespace, the catalog is the source of truth for object dense vectors in that namespace; BYOE writes for the same `(namespace, object id)` are rejected (not silently overwritten) for clarity. Namespaces with the feature disabled continue to honor BYOE writes unchanged. (This is the policy chosen to satisfy FR-018.)
- **Re-embed cadence**: Namespace-wide re-embed is operator-triggered, not automatic. Detection of a strategy/version change does not automatically initiate a re-embed — the operator must explicitly opt in, since the cost is real (CPU time in V1, and additionally external billing for future LLM strategies).
- **Storage of raw content**: V1 stores raw catalog content in the existing primary database alongside other operational records (events, namespace configs, batch run logs). Migrating raw content to dedicated object storage is a possible follow-up if scale justifies it but is not part of V1 scope.
- **Dead-letter handling**: Items that exhaust retries are retained, not deleted. Operators must be able to inspect them and re-drive them; they do not silently disappear.
- **Resource reporting granularity**: Throughput / resource reporting is per-namespace and accumulated continuously. Per-item attribution is out of scope for V1. Future external-LLM strategies may additionally report monetary cost through the same surface.
- **Recommend-side compatibility**: The existing recommend service requires no change. Catalog-produced dense vectors flow into the same dense object collection that BYOE writes target today, so the existing hybrid-blend logic applies unchanged.
- **Authentication continuity**: Catalog ingest reuses the existing per-namespace API key authentication scheme. No new auth surface is introduced for clients.
