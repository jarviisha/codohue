# Implementation Plan: Admin Pipeline Controls

**Branch**: `feat/web-admin-dashboard` | **Date**: 2026-05-03 | **Spec**: [spec.md](spec.md)  
**Input**: Feature specification from `specs/002-admin-pipeline-controls/spec.md`

## Summary

Add three admin capabilities to the Codohue web dashboard: (1) on-demand batch run trigger per namespace, (2) paginated events listing with optional subject filter, and (3) test event injection from the UI. Together these close the feedback loop for pipeline development тАФ inject тЖТ batch тЖТ inspect recommendations тАФ without waiting for the cron interval.

Backend: 3 new admin API endpoints in `internal/admin/`, export `compute.Job.processNamespace` as `RunNamespace`, wire a `*compute.Job` into the admin service for synchronous batch execution. Frontend: new `EventsPage`, three new React Query hooks, "Run now" button on namespace cards.

## Technical Context

**Language/Version**: Go 1.23 (backend), TypeScript / React 19 (frontend)  
**Primary Dependencies**: chi router, pgxpool, go-redis, Qdrant gRPC client (backend); TanStack React Query v5, React Router v7, Tailwind CSS v4 (frontend)  
**Storage**: PostgreSQL `events` table (existing, indexed on namespace + occurred_at)  
**Testing**: `go test ./...` + `make test-pkg` (backend), `tsc --noEmit` (frontend)  
**Target Platform**: Linux server (admin process), Desktop browser (frontend)  
**Project Type**: Internal SaaS admin dashboard (web service + SPA)  
**Performance Goals**: Events list <1s for namespaces with up to 1M events (server-side pagination); batch trigger completes in same wall time as cron-triggered batch  
**Constraints**: Batch trigger synchronous (blocks request); 10-minute context timeout; no new DB migrations  
**Scale/Scope**: Single-tenant admin tool; one concurrent batch trigger per namespace at a time

## Implementation Phases

### Phase A тАФ Export RunNamespace (compute)

**File**: `internal/compute/job.go`

Rename `processNamespace` тЖТ `RunNamespace` (exported). Update the single internal caller in `runOnce`. No behavioral change.

```go
// RunNamespace runs all batch phases for a single namespace synchronously.
func (j *Job) RunNamespace(ctx context.Context, ns string) {
    // body unchanged from processNamespace
}
```

**Why exported instead of a new wrapper**: The method already does exactly what the admin needs тАФ log insertion, all three phase orchestrations, log update. An empty wrapper would violate the "no abstraction beyond what the task requires" principle.

---

### Phase B тАФ New Types (admin/types.go)

Add four new types (see `data-model.md` for exact field definitions):
- `TriggerBatchResponse`
- `EventSummary`
- `EventsListResponse`
- `InjectEventRequest`

---

### Phase C тАФ Repository (admin/repository.go)

Add `GetRecentEvents(ctx, ns, limit, offset int, subjectID string) ([]EventSummary, int, error)`.

Implementation: two sequential reads (no transaction needed тАФ both are reads):
1. COUNT with namespace + optional subject_id filter
2. SELECT with LIMIT/OFFSET ordered by `occurred_at DESC`

---

### Phase D тАФ Service (admin/service.go)

**Struct additions**:
```go
type Service struct {
    // ... existing fields unchanged ...
    job          *compute.Job
    runningBatch sync.Map  // keyed by namespace name, tracks in-flight batch runs
}
```

`NewService` gains a `job *compute.Job` parameter.

**Three new methods**:

**`TriggerBatch(ctx, ns)`**:
1. `GetNamespace` to confirm namespace exists тЖТ 404 if not
2. `runningBatch.LoadOrStore(ns, true)` тЖТ 409 if already loaded
3. `defer runningBatch.Delete(ns)`
4. Apply 10-minute context timeout
5. Call `job.RunNamespace(ctx, ns)`
6. Read back the latest batch log row to build `TriggerBatchResponse`

**`GetRecentEvents(ctx, ns, limit, offset, subjectID)`**:
1. Clamp: `if limit <= 0 { limit = 50 }; if limit > 200 { limit = 200 }`
2. `repo.GetRecentEvents(...)` тЖТ `*EventsListResponse`

**`InjectEvent(ctx, ns, req)`**:
1. Set `occurred_at` to `time.Now().UTC()` if not provided
2. Build payload JSON
3. HTTP POST to `{apiURL}/v1/namespaces/{ns}/events` with Bearer auth (`s.apiKey`)
4. Return nil on 202, wrapped error on other status codes

---

### Phase E тАФ Handler (admin/handler.go)

Three new handler methods following the existing style:

**`TriggerBatch(w, r)`**: Extract `{ns}`, call service, map errors (404тЖТ404, "running"тЖТ409, deadlineтЖТ504, elseтЖТ500), write JSON.

**`GetRecentEvents(w, r)`**: Parse `?limit`, `?offset`, `?subject_id`, validate ranges, call service, write JSON.

**`InjectEvent(w, r)`**: Decode body, validate non-empty `subject_id` and `object_id`, call service, write 202 `{"ok": true}`.

---

### Phase F тАФ Route Registration (cmd/admin/main.go)

1. Import and initialize `*compute.Job` the same way `cmd/cron/main.go` does (shared infra: postgres pool, redis, qdrant, nsconfig service, compute service)
2. Pass `job` to `admin.NewService(..., job)`
3. Register inside the protected group:
   ```go
   r.Post("/api/admin/v1/namespaces/{ns}/batch-runs/trigger", h.TriggerBatch)
   r.Get("/api/admin/v1/namespaces/{ns}/events", h.GetRecentEvents)
   r.Post("/api/admin/v1/namespaces/{ns}/events", h.InjectEvent)
   ```
4. Update CLAUDE.md REST API table

---

### Phase G тАФ Frontend Hooks

**`useTriggerBatch.ts`** тАФ `useMutation`:
- POST `/api/admin/v1/namespaces/{ns}/batch-runs/trigger`
- On success: invalidate `['batch-runs']` and `['namespaces-overview']` queries

**`useEvents.ts`** тАФ `useQuery`:
- GET `/api/admin/v1/namespaces/{ns}/events?limit=&offset=&subject_id=`
- Enabled only when `namespace !== ''`
- `staleTime: 5_000`

**`useInjectEvent.ts`** тАФ `useMutation`:
- POST `/api/admin/v1/namespaces/{ns}/events` with `InjectEventRequest` body
- On success: invalidate `['events', ns]` query

---

### Phase H тАФ EventsPage

**File**: `web/admin/src/pages/EventsPage.tsx`

Layout (top тЖТ bottom):
1. **Page header** тАФ "Events" title + namespace selector dropdown
2. **Inject form** тАФ always visible; fields: Subject ID (text), Object ID (text), Action (select from namespace's action_weights keys or default set), Submit button
3. **Filters row** тАФ Subject ID filter input + "Apply" button
4. **Events table** тАФ columns: Time | Subject ID | Object ID | Action | Weight
5. **Pagination** тАФ "тЖР Prev / Next тЖТ" with "Showing XтАУY of Z total"

All styling uses DESIGN.md semantic token classes (no hardcoded hex).

Empty state: "No events found. Use the inject form above or send events via the main API."
